# RAGAS: метрики и как их улучшать

## Что такое RAGAS

RAGAS (Retrieval Augmented Generation Assessment) — фреймворк для автоматической оценки RAG-пайплайнов. Не требует размеченных пар вопрос→ответ для большинства метрик — судья-LLM сам выносит вердикт.

Каждая метрика оценивает свой слой пайплайна:

```
вопрос
  └→ retriever → [чанк1, чанк2, ...] → generator → ответ
                     ↑                      ↑
           context_precision          faithfulness
           context_recall             answer_relevancy
```

---

## Метрики

### `faithfulness` — честность ответа

**Что измеряет:** насколько ответ опирается только на найденные чанки, не добавляя ничего от себя.

**Как считается:**
1. LLM разбивает ответ на атомарные утверждения
2. Каждое утверждение проверяется: есть ли оно в контексте?
3. `score = подтверждённых_утверждений / всего_утверждений`

**Что значит низкий score:** модель галлюцинирует — генерирует факты, которых нет в чанках. Либо чанки настолько бедны, что модель вынуждена додумывать.

**Как растить:**
- Улучшить retrieval (дать модели нужные чанки) — тогда ей не нужно додумывать
- Добавить в system prompt явный запрет отвечать вне контекста: _"Отвечай только на основе предоставленных фрагментов. Если информации нет — скажи об этом."_
- Уменьшить temperature генератора
- Попробовать более послушную модель-генератор (модели разного размера галлюцинируют по-разному)

---

### `answer_relevancy` — релевантность ответа

**Что измеряет:** насколько ответ по существу вопроса. Ловит уклончивые, размытые, офтопичные ответы.

**Как считается:**
1. LLM генерирует N "обратных вопросов" из ответа — какой вопрос мог бы породить этот ответ?
2. Обратные вопросы и оригинальный вопрос эмбеддятся
3. `score = среднее cosine_similarity(оригинал, обратные_вопросы)`

Если ответ уклончивый ("это сложная тема, есть разные мнения...") — обратные вопросы будут далеко от оригинала → низкий score.

**Что значит низкий score:** ответ есть, но не на тот вопрос. Часто причина — плохой контекст: модель получила нерелевантные чанки и сгенерировала что смогла.

**Как растить:**
- Улучшить retrieval (правильные чанки → правильный ответ)
- Добавить в system prompt: _"Отвечай конкретно на заданный вопрос, без лишних отступлений"_
- Уменьшить TOP_K — меньше мусорных чанков в контексте
- Проверить качество вопросов в eval датасете: размытый вопрос → размытый ответ → низкий score

---

### `context_precision` — точность контекста

**Что измеряет:** насколько найденные чанки действительно нужны для ответа. Ловит "мусор" в топе выдачи.

**Как считается:**
1. Для каждого чанка LLM спрашивает: полезен ли он для ответа на вопрос?
2. Вычисляется precision@K с весами по позиции (чанк на 1-м месте важнее чанка на 5-м)
3. `score` падает, если нерелевантные чанки стоят высоко в списке

Требует `reference` (эталонный ответ) в датасете.

**Что значит низкий score:** retriever тащит много нерелевантных чанков, особенно в начало списка. Либо чанки слишком большие и содержат и нужное, и ненужное.

**Как растить:**
- Уменьшить размер чанков (`CHUNK_SIZE`) — меньше "размытия" по нерелевантному тексту
- Добавить reranker (cross-encoder) — переранжирует топ после retrieval
- Поднять порог отсечения по score/distance — не брать слабые совпадения
- Улучшить embedding-модель (релевантные чанки должны быть геометрически ближе к запросу)
- Уменьшить TOP_K — жёсткий отбор

---

### `context_recall` — полнота контекста

**Что измеряет:** все ли нужные факты попали в найденные чанки. Ловит случаи когда retriever "не нашёл" нужное.

**Как считается:**
1. LLM разбивает `reference` на атомарные факты
2. Для каждого факта проверяет: есть ли он в `retrieved_contexts`?
3. `score = найденных_фактов / всего_фактов_в_эталоне`

Требует `reference` в датасете.

**Что значит низкий score:** нужные чанки вообще не попали в топ. Retriever их просто не нашёл.

**Как растить:**
- Увеличить TOP_K (брать больше чанков)
- Hybrid search — BM25 поймает то, что упустит вектор
- Улучшить embedding-модель (лучше понимает семантику → лучше recall)
- Уменьшить `CHUNK_SIZE` и увеличить `CHUNK_OVERLAP` — меньше шанс что нужный факт "разрезан" между чанками
- Query expansion: перефразировать запрос несколькими способами, объединить результаты

---

## Зависимости метрик

```
embedding-модель ──→ context_recall ──→ answer_relevancy
                 └→ context_precision ─┘
                                        └→ faithfulness (слабее)
system prompt ────────────────────────→ faithfulness
                                     └→ answer_relevancy
```

Плохой retrieval тянет вниз всё. Поэтому retrieval-метрики (`context_recall`, `context_precision`) — приоритет.

---

## Типичная стратегия улучшения

| Шаг | Что делать | Какие метрики растут |
|-----|-----------|----------------------|
| 1 | Лучше embedding-модель | recall, precision, relevancy |
| 2 | Hybrid search (BM25 + vector) | recall |
| 3 | Reranker (cross-encoder) | precision |
| 4 | Тюнинг чанкинга (размер, overlap) | recall, precision |
| 5 | Query expansion | recall |
| 6 | Тюнинг system prompt | faithfulness, relevancy |
| 7 | Лучше генератор | faithfulness, relevancy |

Шаги 1–4 — retrieval. Шаги 6–7 — generation. Начинать с retrieval: generation не поможет, если контекст плохой.

## Ловушки при интерпретации

**Судья меняет результаты.** Слабый судья (мелкая модель) неправильно парсит цепочки рассуждений → метрики занижены. Для сравнения экспериментов использовать одного и того же судью.

**`faithfulness = 1.0` — не всегда хорошо.** Если retriever нашёл нерелевантные чанки, а модель честно пересказала их — faithfulness высокий, но ответ бесполезный. Смотри в связке с `context_recall`.

**`context_precision` чувствителен к порядку.** Метрика взвешена по позиции: нерелевантный чанк на 1-м месте хуже, чем на 5-м. Алгоритм слияния (RRF) влияет на это напрямую.

**Маленький датасет — высокая дисперсия.** 10–15 вопросов дают нестабильные оценки. Минимум для ориентира — 30+ вопросов.
