# ADR-004: Cross-Encoder Reranker в гибридном поиске

**Дата:** 2026-05-12
**Статус:** Принято
**Автор:** Никита Бабченко

---

## Контекст

После внедрения гибридного поиска (BM25 + векторный, RRF-слияние, ADR-002) качество retrieval
ограничено природой обоих методов: BM25 работает на лексическом совпадении, векторный поиск —
на близости эмбеддингов. Оба метода ранжируют документы независимо от запроса как целостной
семантической единицы. RRF усредняет ранги, но не переоценивает пары (query, doc).

Следующий логичный шаг — добавить модель, которая видит запрос и документ **вместе** и выдаёт
скор релевантности, учитывая их взаимодействие (cross-attention).

---

## Решение

Добавить cross-encoder reranker как третий этап retrieval-пайплайна:

```
Все чанки (N)
    → BM25 top-30 + Vector top-30
    → RRF → 30 кандидатов
    → Cross-encoder reranker
    → top-K финальных чанков → LLM
```

**Выбранная модель:** `BAAI/bge-reranker-v2-m3`

- Многоязычная (русский поддерживается)
- Запускается локально через `sentence-transformers` (библиотека уже в проекте)
- Размер: ~570 MB, скачивается один раз с HuggingFace
- Инференс на CPU без GPU

**Реализация:**
- Lazy-загрузка модели (синглтон `_reranker`) — не тормозит старт приложения
- Новая функция `hybrid_retrieve_reranked()` в `retriever.py`
- В UI добавлен режим `hybrid+rerank` (выбран по умолчанию) наряду с `hybrid` и `vector`

---

## Рассмотренные альтернативы

| Вариант | Почему отклонён |
|---|---|
| `cross-encoder/ms-marco-MiniLM-L-6-v2` | Обучен на английском, хуже на русском |
| Cohere Rerank API | Платный API, нет оффлайн-режима |
| `bge-reranker-base` | Менее точный, не многоязычный |
| Увеличить top-K напрямую в hybrid | Больше шума в контексте LLM, выше стоимость |

---

## Последствия

**Положительные:**
- Cross-encoder видит семантическое взаимодействие (query, doc) — ловит случаи, где embedding
  не справляется (вопрос и ответ лексически далеко)
- Нет дополнительных API-расходов — модель локальная, вызывается до LLM
- 30 кандидатов → cross-encoder → top-K снижает шум по сравнению с прямым top-K из hybrid

**Отрицательные / Риски:**
- +200–500 мс на запрос (CPU-инференс на 30 парах)
- Первый запрос после старта медленнее (загрузка ~570 MB модели)
- Увеличение использования RAM (~1 GB под модель)
- BM25-кэш и reranker-синглтон — состояние в памяти процесса; при нескольких воркерах
  каждый держит свою копию
